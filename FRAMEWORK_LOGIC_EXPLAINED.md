# 框架工作逻辑说明

## 这份文档是给谁看的

这份文档面向不想直接阅读训练代码的人，用尽量直白的中文解释这个框架到底在做什么。

## 一句话概括

这个框架想做的事情是：

把

1. 历史时间序列数据
2. 可能影响未来变化的新闻信息

结合起来，用于预测未来的电力相关指标。

它的核心思想是：

- 先只靠历史数据学出“正常情况下的走势”
- 再让新闻去修正“历史模型没有看出来的那部分误差”

可以把它理解成：

- `Base model` = 常规预测器
- `DELTA model` = 修正器

## 这个框架在解决什么问题

如果只看历史电力数据，模型通常能学到很多规律，比如：

- 日内周期
- 工作日和周末差异
- 季节性变化

但现实中还会有很多外部事件影响结果，比如：

- 极端天气
- 电网异常
- 市场波动
- 政策、燃料、供需相关的新闻

这些影响，往往不能只从历史曲线里直接看出来。

所以这个框架并不是让新闻从头开始做预测，而是让：

- 历史模型负责主要趋势
- 新闻模块负责修正特殊事件带来的偏差

## 整体思路

整个框架可以粗略分成两个核心阶段。

### 第一阶段：Base 预测

第一步，框架会先训练一个普通的时间序列模型，只使用历史数值数据。

它的任务很简单：

- 看过去的数据
- 预测未来的趋势

这个模型给出的结果，就是“基础答案”。

用最直白的话说，就是：

“如果我完全不看新闻，只靠过去的数值变化，我会怎么预测未来？”

## 第二阶段：基于新闻的修正

有了 Base 预测之后，框架会继续问第二个问题：

“Base 模型哪里预测错了？这些误差能不能由新闻来解释？”

这就是 `DELTA` 阶段。

这里的 `delta`，本质上表示：

- 真实值
- 减去
- Base 模型预测值

也就是“Base 模型还没预测好的那部分误差”。

因此最终预测可以理解为：

`最终预测 = Base 预测 + 新闻修正`

这也是这个框架比“从一开始就把所有信息混在一起训练”的方法更容易理解的原因。

因为它把职责分得很清楚：

- Base 负责正常规律
- DELTA 负责剩余误差

## 新闻是怎么被使用的

这个框架并不是把原始新闻全文直接扔给模型就结束了。
它会对新闻做几层处理。

### 1. 新闻匹配

对于每一个预测样本，框架会去找和该时间点相关的新闻。

### 2. 新闻整理

原始新闻文本可能会被截断、清洗、重组，变成更适合模型使用的形式。

为了避免重复做这些比较耗时的工作，框架会把中间结果缓存下来。

所以你会看到类似这样的文件：

- `checkpoints/_shared_refine_cache/news_doc_cache_news_2024_2025.json`

### 3. 结构化新闻特征

除了文本本身，框架还会尝试把新闻提炼成更规则的结构化特征，比如事件相关信号。

这样模型在使用新闻时会更稳定，也更容易和数值时间序列结合。

## SignNet 是干什么的

在修正预测时，一个关键问题不只是“修正多少”，还有“修正方向是什么”。

例如：

- 应该往上修正吗？
- 还是应该往下修正？

框架里有一个辅助模块叫 `SignNet`，它主要负责这件事。

你可以把它理解成一个“方向判断器”。

它的作用是：

- 学习这次修正应该是正向还是负向
- 给 DELTA 提供更稳定的方向信息

用更通俗的话说：

- Base 模型说：“我觉得明天大概是 X”
- SignNet 说：“结合新闻看，修正方向更像是往上”或者“往下”
- DELTA 再决定具体修正多少

## 一个很直观的例子

假设系统要预测电力负荷。

如果没有新闻：

- Base 模型可能会预测这是普通的一天

但这时出现了新闻：

- 极端高温
- 空调负荷可能明显增加

那么这个框架可能会这样工作：

1. Base 模型先给出一个正常日的负荷预测。
2. 新闻模块发现有高温事件。
3. SignNet 判断修正方向应该偏正，也就是往上调。
4. DELTA 估计应该额外增加多少负荷。
5. 最终预测值高于原始的 Base 预测。

所以这个系统不是在替代 Base 模型，而是在 Base 模型的基础上做更有针对性的修正。

## 一次完整运行会发生什么

下面用尽量口语化的方式，说明一次完整训练的大致流程。

### 步骤 1：读取数据

框架会读取：

- 训练集 / 验证集 / 测试集的时间序列数据
- 新闻数据

输入：

- 原始时间序列文件，例如带有时间列和目标值列的表格
- 原始新闻文件，例如带有新闻时间和正文内容的表格或 JSON
- 任务配置，例如：
- 目标列是什么，比如 `TOTALDEMAND` 或 `RRP`
- 时间列是什么，比如 `date`
- 历史窗口长度 `history_len`
- 预测长度 `horizon`
- 数据划分文件路径、步长、batch size 等

输出：

- 可直接用于训练的 train / validation / test 划分
- DataLoader 可逐批返回的样本字段，典型包括：
- `history_value`：过去一段时间的目标值序列
- `target_value`：未来要预测的真实值序列
- `history_times`：历史窗口内各时间点
- `target_times`：未来预测区间各时间点
- `target_time`：该样本对应的预测起点
- `series_id`：序列标识
- 解析好的新闻表
- 后续阶段可共享的运行配置和数据统计信息

### 步骤 2：训练 Base 模型

Base 模型只利用历史数值数据，学习正常情况下的变化规律。

输入：

- 训练集中的 `history_value`
- 与之对应的 `target_value`
- Base 模型结构，例如当前脚本里常见的 `mlp`
- Base 阶段超参数，例如：
- `history_len`
- `horizon`
- `base_epochs`
- `base_lr`
- `base_hidden_dim`
- `base_dropout`
- 训练时使用的标准化统计量，例如全局均值和标准差

输出：

- 一个已经学会“只看历史数据做预测”的 Base 模型
- 每个 epoch 的训练过程与验证指标
- 多个候选 checkpoint
- 当前最好的一份 Base 权重
- 一个“历史模式基线”，供后续 DELTA 阶段作为参照

### 步骤 3：评估 Base 模型

框架会在验证集上检查 Base 模型效果，并保存表现最好的 Base checkpoint。

输入：

- 当前 Base 模型参数
- 验证集中的历史窗口和目标窗口
- 评价规则，例如：
- `val_loss(zMSE)`
- `val_mse(raw)`
- `val_mae(raw)`
- “以哪个指标作为最好模型”的选择规则

输出：

- 当前 Base 模型在验证集上的误差指标
- 最佳 Base checkpoint 路径
- 一个明确的基准成绩，表示：
- 如果完全不使用新闻，模型能做到什么水平
- 这个基准也会在 DELTA 阶段用来比较“新闻修正到底有没有帮助”

### 步骤 4：准备新闻特征

框架会把新闻和每个预测样本对齐，并且可能会：

- 整理文本
- 读取或建立缓存
- 提取结构化新闻特征

输入：

- 当前样本的 `target_time`
- 该时间点附近候选新闻
- 新闻筛选参数，例如窗口大小、最多取多少条新闻
- 新闻整理配置，例如是否启用 API refine、是否启用结构化抽取
- 已有缓存文件，例如：
- refined news cache
- structured news cache
- tokenizer 和最大长度设置

输出：

- 与该样本真正匹配到的新闻集合
- 整理后的新闻文本，例如合并后的 refined news
- 按文章保留的 refined doc 列表
- 结构化事件与结构化特征向量，例如 `structured_feats`
- 新闻条数统计，例如 `news_counts`
- 供模型直接使用的文本 token、attention mask、doc 级输入
- 可重复利用的缓存结果，避免下次重复处理同一批新闻

### 步骤 5：训练 SignNet

在正式训练 DELTA 之前，框架可以先训练 SignNet，让修正方向判断更稳定。

输入：

- `history_raw`
- 它具体就是当前样本预测起点之前过去 `L` 步的原始目标值序列
- 在代码里来自 `history_value`
- `base_pred_z`
- Base 模型对未来 `H` 步的预测结果
- `targets_z`
- 未来 `H` 步真实目标值的标准化结果
- `structured_feats`
- 由新闻提炼出来的结构化特征向量
- `news_counts`
- 当前样本匹配到的新闻数量
- 这些信息组合起来后，SignNet 要学习的是：
- 未来每个预测位置上的残差方向是正还是负

输出：

- 一个训练好的方向分类器 `SignNet`
- horizon 维度上的方向 logits
- soft sign 信号，也就是“往上修正还是往下修正”的连续方向信息
- SignNet 的验证指标，例如：
- `val_loss`
- `val_acc`
- `val_bacc`
- 训练完成后可供 DELTA 调用的方向判断能力
- 如果开启校准，还会得到校准后的方向偏置

### 步骤 6：训练 DELTA

DELTA 模型学习的是“Base 模型剩下的误差”，也就是：

- 哪些地方 Base 预测偏低了
- 哪些地方 Base 预测偏高了
- 什么时候新闻应该起作用
- 什么时候新闻应该被忽略

输入：

- 最佳 Base checkpoint
- 每个样本的 Base 预测结果
- 时间序列输入的 patch 表示，例如 `ts_patches` 和 `ts_patch_mask`
- 真实目标值的标准化结果 `targets_z`
- 整理后的新闻文本输入，例如：
- `refined_news_ids`
- `refined_news_attn`
- `refined_news_doc_ids`
- `refined_news_doc_attn`
- `refined_news_doc_mask`
- 结构化新闻特征，例如 `structured_feats`
- 新闻数量、相关性标签等辅助信息，例如 `news_counts`、`rel_labels`
- 可选的外部方向信号，也就是 SignNet 产出的 sign 信息
- DELTA 阶段的各种约束与损失配置，例如：
- 非退化约束
- gate 监督
- sign 监督
- magnitude 监督
- 文本分支 / 结构化分支开关

输出：

- 一个训练好的 DELTA 修正模型
- 模型内部学到的修正机制，例如：
- 什么时候该开 gate
- 修正方向如何选择
- 修正幅度应该多大
- DELTA 验证指标，例如验证集上的 MAE、与 Base 的对比改进等
- 最佳 DELTA checkpoint
- 如果启用了外部 SignNet，也会把相应方向模块一起保存下来
- 一套建立在 Base 预测之上的“误差修正能力”

### 步骤 7：合并 Base 和 DELTA

框架会把：

- Base 预测
- DELTA 修正

组合起来，得到最终预测结果。

输入：

- Base 预测值
- DELTA 预测出的修正项
- 内部控制信号，例如：
- gate
- sign
- magnitude
- 各类融合开关与裁剪参数
- 当前所使用的残差模式，例如 additive 模式

输出：

- 最终修正后的预测值
- 对每个预测位置的最终融合结果
- 一个可以直接与真实值比较的最终预测序列

### 步骤 8：验证与早停

在训练过程中，框架会不断看验证集表现，并在改进变得很小的时候提前停止。

输入：

- 当前 epoch 的模型参数
- 当前 epoch 在验证集上的误差指标
- 最优历史成绩
- 早停相关参数，例如 patience、min_delta、选择指标
- 对 DELTA 来说，还会比较它是否真正优于 Base 基线

输出：

- “继续训练”或“停止训练”的决定
- 当前是否刷新最佳 checkpoint
- 一份稳定保留下来的最佳模型状态
- 一条清晰的训练轨迹，说明模型是继续改善还是开始停滞

### 步骤 9：测试并保存结果

训练结束后，框架会写出：

- checkpoint
- 日志
- 图像
- 测试指标

输入：

- 保存下来的最佳 Base 或 DELTA checkpoint
- 测试集中的历史窗口、目标窗口、时间信息
- 训练过程中记录的标准化信息
- 需要写出的结果路径和日志路径

输出：

- 最终测试指标，例如：
- `loss(zMSE)`
- `mse(raw)`
- `mae(raw)`
- 预测值与真实值对比图
- 保存到 `checkpoints/` 下的模型、日志和图像
- 保存到 `results/` 下的结果汇总，例如 `test_results.csv`
- 一个可复盘的实验结果记录，方便后续对比不同配置

## 为什么要这么设计

这个设计有几个很实际的好处。

### 优点 1：更容易排查问题

你可以分别看清楚：

- Base 预测本身做得怎么样
- 新闻到底有没有帮助
- 修正方向是否合理

### 优点 2：比“让新闻控制一切”更稳定

新闻本身是有噪声的。
如果模型过度依赖新闻，很容易变得不稳定。

这个框架把 Base 设为主预测器，让 DELTA 只做修正，因此整体会更容易控制。

### 优点 3：更适合处理突发事件

纯时间序列模型通常在“正常时期”表现不错，但遇到突发事件时可能会明显变弱。
这个框架正是想补上这块短板。

## 主要文件分别代表什么

如果你想快速认识这个项目，可以先把下面几个文件和目录当作地图来看。

- `run.py`
  主程序入口。负责接收命令行参数并启动训练。

- `src/base_delta_decoouple_trainer.py`
  核心训练流程所在文件。负责 Base 训练、DELTA 训练、评估和保存结果。

- `scripts/nswelecLOAD_2024_tinynews.sh`
  具体任务的启动脚本。里面定义了数据集路径、模型设置、预测 horizon 等实验配置。

- `checkpoints/`
  用来保存模型 checkpoint 和运行日志。

- `results/`
  用来保存最终结果汇总，比如测试指标。

## 日志应该怎么读

如果你在看训练日志，可以用下面这种方式快速理解当前框架在做什么：

- `[BASE]`
  说明框架正在训练或评估只看历史数据的基础模型。

- `[SIGNNET]`
  说明框架正在学习修正方向。

- `[DELTA]`
  说明框架正在学习基于新闻的误差修正。

- `[TEST][FINAL]`
  说明训练已经走到最终测试阶段，并且在写最终结果。

## 最容易记住的心智模型

如果你只想记住一个最简单的理解方式，可以把这个框架想成一个“两个人合作”的系统。

- 第一个人只看历史电力曲线，给出一个常规预测。
- 第二个人去看新闻，然后说：“今天可能有特殊情况，应该对这个预测做调整。”

最终答案，就是这两个人合作后的结果。

## 用最通俗的话总结

这个框架并不是让新闻从零开始预测电力指标。

它真正做的是：

- 先相信历史数据给出的主趋势
- 再利用新闻解释特殊事件
- 最后做一个谨慎的修正，而不是完全推翻原来的预测

这就是这个框架的核心工作逻辑。
